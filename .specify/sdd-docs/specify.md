# 專案需求規格 (Specify Phase)

## 全域系統需求
- **視覺與體驗**：網站需具備現代化美學設計，且必須完整支援響應式設計（Responsive Design），確保在桌面、平板與行動裝置上皆有流暢的使用體驗。
- **存取控制**：前台頁面全面公開；後台管理頁面需具備權限驗證機制，未經驗證之使用者應被阻擋並導向至首頁。

---

## 1. 專案首頁 (Public Gallery View)

### 1.1 視覺焦點區塊 (Hero Section)
- **核心目標**：在使用者進入網站的第一時間，快速傳達網站主題與核心描述。
- **功能需求**：需包含吸睛的視覺引導，並整合動態的畫廊跑馬燈（或自動輪播區塊），以動態方式初步展示精選圖片。

### 1.2 作品集展示區塊 (Gallery Grid)
- **核心目標**：提供使用者流暢的圖片瀏覽體驗，並能檢視詳細的圖片生成資訊（如提示詞與標籤）。
- **資料呈現規則**：
  - **可見性限制**：僅查詢並展示狀態為「啟用」（`isActive: true`）的圖片項目。
  - **排序機制**：資料需依據建立時間（`created_at`）由最新至最舊（Descending）排列。
  - **載入機制**：必須實作滾動動態載入（Infinite Scroll）或分頁，以確保大量圖片呈現時的效能。需具備圖片載入中的佔位符（Skeleton）與載入失敗的視覺處理。
- **使用者互動**：
  - **列表瀏覽**：需以清晰的網格排版展示圖片縮圖，並附帶標題（`title`）與分類標籤（`badges`）。標籤需能依據設定呈現不同顏色以區分層級或屬性。
  - **細部資訊檢視**：點擊單張縮圖後，需以彈出視窗或滿版覆蓋的方式展示細節。內容必須包含：高解析度圖片（`image_url`）、標題、完整提示詞（`prompt`）、標籤陣列、建立時間。
  - **提示詞操作**：在細節視窗中，必須提供「複製提示詞（Copy Prompt）」的快捷按鈕。使用者點擊後需將提示詞寫入剪貼簿，並立即給予「複製成功」的視覺反饋。
- **邊界與例外處理**：
  - **空狀態提示**：當系統中沒有任何狀態為啟用的圖片時，必須顯示明確且友善的空狀態畫面（例如：「目前尚無公開的展示作品」）。

## 2. 管理者頁面 (Admin Dashboard)

### 2.1 全域資料管理列表
- **核心目標**：提供管理者檢視與操作所有 `GalleryItem` 資料的集中式介面。
- **功能需求**：
  - 需呈現資料庫中所有的圖片紀錄（包含 `isActive` 為 `true` 與 `false` 的項目）。
  - 列表需清晰展示每筆資料的關鍵屬性，並在每列資料後方提供「編輯」與「刪除」的操作入口。

### 2.2 新增與編輯表單
- **核心目標**：讓管理者能順暢地新增作品或修改現有作品的屬性與狀態。
- **介面行為**：點擊新增或編輯時，需以側邊滑出（Slideover）或疊層的方式開啟表單，保持列表頁面的上下文不中斷。
- **標籤 (Badges) 管理**：
  - 表單內的標籤欄位需以列表或表格形式呈現。
  - 必須提供新增標籤的按鈕，且每一列既有標籤皆須配備獨立的刪除按鈕。
- **自訂驗證邏輯 (Custom Validation)**：
  - 表單送出前必須進行嚴格的自訂驗證，且**「新增 (Create)」與「編輯 (Edit)」模式必須套用不同的驗證規則**（例如：新增時圖片檔案為絕對必填；編輯時若未上傳新圖片，則略過該欄位的必填驗證）。
- **阻斷與防呆機制 (Critical)**：
  - **圖片網址獲取檢核**：在表單提交（Submit）的流程中，系統必須向儲存服務請求圖片的公開網址。**若無法成功獲取公開網址（Public URL），表單提交必須失敗並立即阻斷後續的資料庫寫入動作**，同時向管理者顯示明確的錯誤原因。

### 2.3 操作回饋與安全性防護
- **刪除防護**：執行任何資料刪除動作前，系統「必須」彈出二次確認視窗，警示管理者，以防止誤觸。
- **狀態通知**：無論是新增、更新或刪除動作，在 API 請求完成後，都必須於畫面上方或角落顯示明確的「成功」或「失敗」通知訊息（Toast/Alert），確保管理者清楚當前操作的結果。
